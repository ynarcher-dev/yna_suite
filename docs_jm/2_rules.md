# 2. 서비스 개발 주의 사항 및 필수 규칙

이 문서는 개발팀이 Y&A 서비스를 구축하고 유지보수할 때 **반드시 지켜야 하는 핵심 업무 규칙**입니다.

---

## 1. 화면(UI) 재사용 원칙
*   **규칙**: 버튼, 입력창, 목록 테이블, 모달(팝업) 등 화면에 그리는 모든 요소는 **임의로 새로 만들지 않고 정해진 표준 컴포넌트를 사용**합니다.
*   **이유**: 서비스마다 버튼 색이나 테이블 디자인이 달라지면 사용자가 혼란을 느끼고 개발 효율성도 극도로 저하됩니다. 
*   **실천**:
    *   새로운 화면을 만들기 전, 공통 UI 창고(`packages/ui`)에 표준 디자인이 이미 구현되어 있는지 먼저 확인합니다.
    *   두 개 이상의 서비스(예: Work와 Fund)에서 동시에 쓰는 컴포넌트는 반드시 공통 UI 창고로 옮겨 관리합니다.

---

## 2. 특이사항 발생 시 기록 의무
*   **규칙**: 개발 도중 예상치 못한 버그, 임시 땜질용 코드(Hotfix), 권한 강제 조작, 예외적 데이터 수정 등이 발생하면 **반드시 이력 메모(`4_memo.md`)에 남겨야 합니다.**
*   **기록 항목**:
    *   누가, 언제, 왜 변경했는지
    *   어떤 데이터나 코드 파일에 영향을 주었는지
    *   향후 어떻게 정식으로 수정할 것인지

---

## 3. 회사의 핵심 색상 준수
*   **규칙**: 서비스 화면은 기본적으로 **차분한 회색조(Grayscale)**를 주색상으로 사용합니다. 회사의 대표 붉은색(`#E22213`)은 **오직 강조할 곳에만 제한적**으로 사용합니다.
*   **이유**: 붉은색을 너무 넓게 사용하면 사용자의 눈이 피로해지고 중요한 정보(예: 경고, 오류 메시지)가 묻히게 됩니다.
*   **실천**:
    *   붉은색 사용처: 주요 저장 버튼, 활성화된 메뉴 탭, 중요 배지, 알림 아이콘, 포커스 테두리.
    *   배경이나 큰 바(Bar) 영역에는 절대 붉은색을 쓰지 않습니다.

---

## 4. 데이터베이스 직접 수정 절대 금지
*   **규칙**: 실제 운영 중인 데이터베이스(DB) 서버에 접속하여 **명령어로 데이터를 임의 수정하거나 삭제하지 않습니다.**
*   **이유**: 수동으로 DB를 만지다 실수하면 데이터가 영구 유실되거나 권한 관계가 깨져 서비스가 멈출 수 있습니다.
*   **실천**:
    *   모든 DB 구조 변경은 사전에 약속된 정식 코드 파일(Migration)을 통해서만 검증 후 배포해야 합니다.
    *   부득이하게 직접 데이터를 정정해야 하는 경우, 반드시 백업 스냅샷을 먼저 확보하고 변경 전후 내용을 기록으로 남깁니다.

---

## 5. 개인정보 마스킹(별표 처리) 의무
*   **규칙**: 사용자 목록, 이메일, 전화번호, 대표자 정보 등 개인 식별이 가능한 정보는 **목록 화면에서 기본적으로 별표(Masking) 처리**합니다.
*   **실천**:
    *   전화번호: `010-****-5678` 형태로 중간 자리를 가립니다.
    *   이메일: `hong***@example.com` 형태로 아이디 뒤쪽을 가립니다.
    *   상세 권한이 있는 담당자만 '상세 보기' 클릭 시에 감사 로그를 남기고 원본 정보를 볼 수 있습니다.

---

## 6. 단위 작업 진행 상황 기록 (기기 간 이어가기)
*   **규칙**: 개발은 **단위 작업 단위로 지시하고 auto mode로 진행**하며, **각 단위 작업이 끝날 때마다 진행 상황을 저장소(git)에 커밋되는 파일로 기록**합니다. 개발 도구(Claude)의 로컬 메모리에만 의존하지 않습니다.
*   **이유**:
    *   작업 노트북이 두 대이고, 어느 기기에서든 이어서 개발할 수 있어야 합니다. 개발 도구의 로컬 메모리는 **기기 간에 공유되지 않으므로**, 진행 기록이 로컬에만 있으면 다른 노트북에서 복구할 수 없습니다. 따라서 기록은 반드시 git으로 동기화되는 저장소 파일에 남깁니다.
    *   각 단위 작업 완료 후 **대화(세션)를 지우고 새로 이어서** 진행하므로, 이전 맥락이 대화에 남아있지 않아도 저장소 파일만 읽으면 현재 상태를 복원할 수 있어야 합니다.
*   **실천**:
    *   단위 작업이 끝나면 `docs_jm/3_checklist.md`의 해당 항목을 `[x]`로 갱신합니다.
    *   `docs_jm/5_progress.md`(개발 진행 로그)에 아래 항목을 최신순으로 추가합니다.
        *   작업 일자 / 작업 기기
        *   완료한 단위 작업(무엇을 만들었는지)
        *   현재 상태(동작 확인 여부, 남은 이슈)
        *   다음에 이어서 할 작업
        *   주의점(임시 처리, 미완성 부분)
    *   기록 후에는 반드시 **commit & push**하여 다른 기기에서 pull로 이어받을 수 있게 합니다.
    *   **새 작업(새 대화)을 시작할 때는 먼저 `3_checklist.md`와 `5_progress.md`를 읽고 현재 지점을 파악한 뒤 이어갑니다.**
    *   예상치 못한 버그·임시 땜질·예외 처리는 위 2번 규칙대로 `4_memo.md`에도 함께 남깁니다.

## 7. 상세 개발 및 코드 작성 규칙 준수
* **규칙**: 실제 코드 구현 시에는 [docs/yna_suite_maintenance_rules.md](file:///c:/dev/yna_suite/docs/yna_suite_maintenance_rules.md)에 기술된 코드 유지보수 및 파일 크기 제한 규칙을 반드시 준수합니다.
* **주요 코드 크기 제한**:
  * 단일 수동 작성 파일: **500줄 이하**
  * 단일 React 컴포넌트: **250줄 이하**
  * 단일 함수: **50줄 이하**
  * 단일 Hook: **150줄 이하**
* **실천**:
  * 파일이나 컴포넌트가 위 기준을 초과하는 경우, 임의로 코드를 계속 덧붙이지 않고 **Hook, Component, Actions, Columns, Schema, Service/Helper** 등으로 로직을 분리합니다.
  * 자세한 분리 예시와 패키지 간 의존성 규칙은 [docs/yna_suite_maintenance_rules.md](file:///c:/dev/yna_suite/docs/yna_suite_maintenance_rules.md)를 참고하십시오.

## 8. AI/바이브 코딩(Vibe Coding) 협업 및 정합성 규칙
*   **규칙**: AI 에이전트를 이용한 바이브 코딩 시 발생할 수 있는 컨텍스트 누수, 코드 비대화, 설계 괴리 등의 부작용을 방지하기 위해 **다음 5가지 수칙을 철저히 준수**합니다.
*   **실천**:
    1.  **시작 진입점 준수**: AI는 새 작업을 시작할 때 반드시 [0_CLAUDE.md](file:///c:/dev/yna_suite/docs_jm/0_CLAUDE.md)를 진입점으로 삼아 현재 Phase와 관련 설계 문서를 먼저 로드하고 시작해야 합니다.
    2.  **체크리스트 및 진척도 실시간 동기화**: 매 단위 작업 완료 시 [3_checklist.md](file:///c:/dev/yna_suite/docs_jm/3_checklist.md)와 [5_progress.md](file:///c:/dev/yna_suite/docs_jm/5_progress.md)를 즉시 업데이트하고 커밋하여, 대화 세션이 초기화되거나 기기를 이동하더라도 AI가 맥락을 즉시 이어받을 수 있게 합니다.
    3.  **임의 의존성(Package) 추가 금지**: 유틸이나 라이브러리가 필요할 때 기존 패키지를 재사용해야 하며, 추가적인 외부 패키지 설치는 반드시 작업자(인간)의 사전 승인을 거쳐야 합니다.
    4.  **DB 스키마 마이그레이션 원칙 고수**: 로컬 및 클라우드 DB 수정 시 콘솔 직접 수정이나 임의 SQL 실행을 금지하고, 오직 `supabase migration` 파일 생성 및 적용을 통해서만 반영합니다.
    5.  **설계 문서 최신화 (Documentation Drift 방지)**: 개발 과정에서 스키마 변경, API 스펙 수정 등이 발생하면 코드 반영과 동시에 `docs/` 폴더 내 관련 설계 문서 내용도 반드시 함께 갱신하여 문서가 낡지 않게 유지합니다.
