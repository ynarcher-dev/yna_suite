# Y&ARCHER WORKS 데이터 품질 및 마스터 거버넌스 가이드

본 문서는 HUB 섹션 마스터 데이터의 품질 관리, 중복 후보 운영, 병합 검토 책임, 데이터 steward 역할, 품질 지표와 운영 주기를 정의한다.

Y&ARCHER WORKS에서 Hub는 단순 목록이 아니라 전사 공통 자원에 대한 신뢰 가능한 마스터 원장이다. 따라서 데이터 품질은 기능이 아니라 운영 체계로 관리해야 한다.

관련 문서:

```txt
마스터 정책: yna_suite_master_data_policy.md
데이터 모델: yna_suite_data_model.md
마이그레이션: yna_suite_migration_strategy.md
DB 운영: yna_suite_database_operations.md
보안 정책: yna_suite_security_policy.md
```

## 1. 핵심 원칙

데이터 품질 관리는 다음 원칙을 따른다.

```txt
입력을 막기보다 사후 정제와 병합 큐로 품질을 높인다.
공식 번호가 없는 상태를 정상 케이스로 본다.
자동 병합보다 운영자 검토를 우선한다.
대표값과 과거값을 분리해 보존한다.
데이터 품질은 담당자와 주기를 정해 운영한다.
```

## 2. 마스터 데이터 대상

1차 관리 대상:

```txt
스타트업/기업
전문가/멘토/평가위원
협력사/기관/LP/자문사
심사역/내부 사용자 프로필
```

관리 항목:

```txt
대표명
공식명
사업자번호/법인번호
대표자명
연락처
이메일
웹사이트
주소
별칭
식별자
유입 출처
검증 상태
병합 상태
```

## 3. 책임 역할

데이터 품질에는 역할이 필요하다.

| 역할 | 책임 |
| :--- | :--- |
| Data Owner | 데이터 정책과 최종 의사결정 |
| Data Steward | 일상적인 품질 검토와 병합 처리 |
| Domain Operator | Work/Fund/M&A 등 도메인 입력 품질 관리 |
| System Admin | 권한, RLS, 운영 설정 관리 |
| Developer | 정규화/중복 후보 로직 구현 |

초기 Phase 1에서는 Hub 관리자와 시스템 관리자가 Data Steward 역할을 겸할 수 있다.

## 4. Verification Status

마스터 검증 상태는 다음을 사용한다.

| 상태 | 의미 |
| :--- | :--- |
| `pending` | 검토 필요 |
| `verified` | 검증 완료 |
| `rejected` | 부적합/잘못된 마스터 |
| `needs_review` | 추가 확인 필요 |
| `temporary` | 임시 유입 상태 |

운영 규칙:

```txt
임시 생성 마스터는 pending 또는 temporary로 시작한다.
공식 식별자 확인 시 verified로 변경할 수 있다.
검증 상태 변경은 audit log 대상이다.
```

## 5. 중복 후보 상태

`hub.merge_candidates.status`는 다음을 사용한다.

| 상태 | 의미 |
| :--- | :--- |
| `pending` | 검토 대기 |
| `approved` | 병합 승인 완료 |
| `rejected` | 별도 대상 확인 |
| `ignored` | 낮은 품질/노이즈 후보 |
| `expired` | 오래되어 재평가 필요 |
| `on_hold` | 추가 확인 보류 |

주의:

```txt
rejected는 같은 후보가 다시 생성되지 않도록 근거를 남긴다.
ignored는 낮은 점수 반복 후보에 사용한다.
on_hold는 외부 확인이 필요한 경우 사용한다.
```

## 6. 중복 후보 우선순위

검토 우선순위:

```txt
1. 강한 식별자 충돌 또는 일치 후보
2. 업무 이력이 많은 임시 마스터
3. 최근 생성된 pending 마스터
4. 점수 80 이상 후보
5. 동일 이름/대표자 후보
6. 낮은 점수 반복 후보
```

점수 기준:

| 점수 | 처리 |
| :--- | :--- |
| 95 이상 | 우선 검토, 자동 연결 가능하나 병합 승인 권장 |
| 80-94 | 운영자 검토 필수 |
| 60-79 | 후보 표시 |
| 60 미만 | 신규 유지 또는 낮은 우선순위 |

자동 병합 금지:

```txt
사업자번호/법인번호 없음
대표자명 충돌
검증된 이메일 충돌
검증된 전화번호 충돌
동명이인 가능성
```

## 7. 병합 검토 기준

병합 승인 전 확인:

```txt
source/target 이름 유사성
공식 식별자 일치 여부
대표자명 일치 여부
전화번호/이메일 일치 여부
웹사이트 도메인 일치 여부
관련 업무 이력
충돌 정보 존재 여부
```

병합 승인 가능:

```txt
강한 식별자가 일치
여러 중간 식별자가 일치
업무 담당자가 동일 대상임을 확인
과거명/법인명 전환 흐름이 명확
```

병합 반려:

```txt
사업자번호가 다름
법인번호가 다름
대표자/연락처가 명확히 다름
동일 이름의 별도 기업/전문가로 확인
```

## 8. 대표값 결정 원칙

병합 시 대표값은 다음 우선순위를 따른다.

공통:

```txt
verified 값 우선
공식 출처 우선
최신 검증값 우선
업무 이력은 union/보존
밀려난 값은 alias/identifier/history로 보존
```

스타트업:

```txt
사업자번호/법인번호: 공식 검증값
법인명: 공식 법인명
팀명/브랜드명: alias
대표자명: 공식 최신값, 과거값 history
산업/태그: union
```

전문가:

```txt
이메일/전화: 검증 primary
소속/직함: 최신 검증값
전문 분야: union
```

협력사:

```txt
기관명: 공식명
사업자번호/법인번호: 검증값
약칭/브랜드명: alias
역할: partner_type 또는 관계 테이블
```

```txt

필수값 누락률

```

권장 대시보드:


## 10. 운영 주기

권장 운영 주기:

| 주기 | 작업 |
| :--- | :--- |
| 매일 | 신규 pending/merge 후보 확인 |
| 매주 | 오래된 후보 처리, 중복률 확인 |
| 매월 | 데이터 품질 리포트 작성 |
| 분기 | 정규화/점수 규칙 개선 |

Phase 1 최소 운영:

```txt
주 1회 pending merge 후보 검토
import 후 batch별 품질 리포트 확인
중요 병합은 사유 기록
```

## 11. SLA 권장 기준

초기 권장:

| 항목 | 기준 |
| :--- | :--- |
| 강한 식별자 후보 | 3영업일 이내 검토 |
| 점수 80 이상 후보 | 5영업일 이내 검토 |
| 일반 pending 후보 | 10영업일 이내 검토 |
| import 실패 row | import 후 5영업일 이내 검토 |

SLA는 팀 규모와 업무량에 따라 조정한다.

## 12. 입력 품질 기준

도메인 앱에서 신규 자원을 입력할 때:

```txt
Hub 검색 먼저 수행
기존 마스터 선택 우선
없으면 임시 마스터 생성
가능한 식별 단서 저장
사용자 업무 흐름은 막지 않음
```

입력 필드 권장:

```txt
스타트업: 회사/팀명, 대표자명, 연락처, 이메일, 사업자번호, 웹사이트
전문가: 이름, 이메일, 전화번호, 소속, 직함, 전문 분야
협력사: 기관명, 사업자번호, 대표자명, 연락처, 기관 유형
```


정규화 변경 시:

```txt

```

예시:

```txt

```

## 14. Import 품질 관리

Import 전:

```txt
원본 백업
컬럼 매핑표 작성
dry-run
필수값 누락 확인
정규화 결과 확인
```

Import 후:

```txt
batch 리포트 확인
신규 마스터 수 확인
기존 연결 수 확인
중복 후보 수 확인
실패 row 검토
품질 이상 패턴 확인
```

품질 경고:

```txt
예상보다 신규 마스터가 과도하게 많음
사업자번호 중복이 많음
전화번호/이메일 형식 오류가 많음
특정 컬럼이 모두 비어 있음
```

## 15. 데이터 정정 요청

정정 요청 처리 흐름:

```txt
1. 요청 접수
2. 대상 마스터 확인
3. 변경 근거 확인
4. 대표값 수정 또는 alias/identifier 추가
5. field_history 기록
6. audit log 기록
7. 요청자에게 결과 공유
```

주의:

```txt
대표값을 바꾸는 대신 alias 추가가 더 적절할 수 있다.
공식 식별자 충돌은 단순 수정하지 않고 검토 대상으로 올린다.
```

## 16. 데이터 품질 권한

권장 권한:

```txt
master:
  전체 병합/정정 가능

hub_write:
  마스터 생성/수정 가능

master_data_merge:
  병합 후보 승인/반려 가능

viewer:
  품질 현황 조회만 가능
```

외부 사용자는 데이터 품질 운영 화면에 접근할 수 없다.

## 17. 금지 사항

```txt
업무 앱별 자체 마스터 테이블 생성
중복 의심 데이터를 임의 삭제
병합 없이 FK만 수동 수정
식별자 원본값을 덮어쓰기
alias/history 없이 대표값만 변경
audit log 없이 병합 승인
공식 번호 충돌 후보 자동 병합
```

## 18. 체크리스트

병합 승인 전:

```txt
강한 식별자가 충돌하지 않는가?
대표자/이메일/전화번호가 모순되지 않는가?
관련 업무 이력을 확인했는가?
병합 후 대표값을 확인했는가?
밀려난 값이 보존되는가?
affected_records가 기록되는가?
사유를 남겼는가?
```

품질 리포트 작성 시:

```txt
pending 후보 수가 줄고 있는가?
임시 마스터가 과도하게 쌓이지 않는가?
import 실패 패턴이 반복되지 않는가?
정규화 규칙 개선이 필요한가?
운영자가 처리할 수 있는 양인가?
```

## 19. 최종 요약

Y&ARCHER WORKS 데이터 품질 운영은 다음을 핵심으로 한다.

```txt
Hub는 전사 마스터 원장이다.
신규 입력은 막지 않고 임시 마스터와 검토 큐로 수용한다.
중복 후보는 점수와 근거로 우선순위를 정한다.
병합은 삭제가 아니라 이력 있는 정리다.
품질은 담당자, 주기, 지표, SLA로 운영한다.
```

좋은 마스터 데이터는 한 번에 깨끗하게 만들어지는 것이 아니라, 입력과 정제와 검토가 반복되는 운영 체계 안에서 유지된다.
